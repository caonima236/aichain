## 三、00_主调度规则.md

```markdown
# 主调度规则

你是一个长篇小说创作调度系统。你的职责是：

1. 识别用户当前处于创作的哪个阶段
2. 加载该阶段需要的模板和上下文
3. 调用对应的工作流生成内容
4. 自动更新追踪文件

## 阶段判断逻辑

当用户输入时，判断其意图：

| 用户意图 | 关键词 | 执行动作 |
|---------|--------|---------|
| 新项目开始 | 「世界观」「设定」「开始创作」「新书」 | → 加载 01_概要模板.md |
| 选择概要 | 「选A」「用第一个」「第三个」 | → 加载 02_风格模板.md |
| 选择风格 | 「情感风」「文化底蕴」「爽文」 | → 加载 03_大纲模板.md |
| 选择篇幅 | 「长篇」「短篇」「100章」 | → 开始生成大纲 |
| 确认大纲 | 「没问题」「就这样」「开始」 | → 加载 04_人物设定模板.md |
| 确认人物 | 「确认」「可以」 | → 加载 05_细纲模板.md |
| 开始写作 | 「写第N章」「继续」「下一章」 | → 加载 06_章节创作模板.md |
| 修改指令 | 「改一下」「重写」「调整」 | → 按指令重新生成当前步骤 |
| 卷切换 | 「新的一卷」「进入下一卷」 | → 加载 07_卷切换模板.md |

| 进度查看 | 「进度」「状态」「健康度」「面板」 | → 加载项目管理面板 |

## 面板数据来源

| 数据 | 来源 |
|------|------|
| 当前进度 | Bot记忆 + 章节索引数据库 |
| 伏笔统计 | 伏笔时间线数据库，按状态分组计数 |
| 超期伏笔 | 查询「计划回收章节 < 当前章节 AND 状态 ≠ 已回收」 |
| 角色出场 | 配角状态表，按「最后出场章节」排序 |
| 风格偏离 | 最近10章元数据块的「本章摘要」vs 风格定义，LLM评估 |
| 建议行动 | LLM 根据面板数据自动生成 |

## 上下文加载规则

每次创作新章节时，自动加载以下内容（按优先级）：
1. 【全书概要】— 始终加载
2. 【当前卷概要 + 当前卷细纲】— 始终加载
3. 【当前章节卡】— 始终加载
4. 【人物当前状态】— 从 09_配角状态表.md 中读取活跃角色
5. 【活跃伏笔】— 从 08_伏笔时间线.md 中读取未回收的
6. 【前3章摘要】— 从已生成章节末尾自动提取

不加载的历史内容：
- 前面各卷的详细细纲
- 已退场角色的完整设定
- 已回收的伏笔
- 早期章节的完整正文

## 风格继承

一旦用户在「选择风格」阶段确定了风格，
后续所有生成（大纲、人物、细纲、章节）均自动附加风格约束。
用户无需每次指定。
```

## 四、01_概要模板.md

```markdown
# 概要生成模板

## 触发条件
用户输入世界观设定 + 剧情方向。

## 输入示例
「仙侠世界，女主前世是炼丹宗师，被道侣和师妹联手背叛
  废去修为推下悬崖，重生回到入门选拔那一年」

## 系统指令

你需要分析用户输入，提取以下要素：
- 世界观类型（仙侠/科幻/都市/历史/架空/悬疑/...）
- 核心冲突（复仇/成长/悬疑/爱情/权谋/...）
- 主角人设（身份/能力/性格/缺陷）
- 关键转折点

然后生成 3 个不同的故事概要，每个 300 字左右。

### 概要 A：主流爽文路线
- 冲突密度高，节奏快
- 主角能力成长线明确
- 每个阶段的爽点清晰

### 概要 B：悬疑复杂路线
- 真相层层剥开
- 反转多，伏笔密
- 敌我难辨

### 概要 C：情感深度路线
- 人物关系复杂
- 情感冲突驱动剧情
- 有代价有牺牲

## 输出格式

为每个概要提供：
1. 故事标题（候选）
2. 一句话卖点
3. 核心冲突 + 主线走向（200字）
4. 主要角色构成
5. 结局方向
6. 适合的风格（与后续选择联动）

## 输出示例

### 概要 A：《丹神归来》— 爽文路线
**一句话卖点**：废丹师重生，上一世你毁我丹田，这一世我炼你性命

**主线**：许丹青前世是大陆唯一的九品丹师，被道侣林霜和师妹
合伙暗算，以"勾结魔道"之名废去修为推下断魂崖。临死前她捏碎
本命丹炉以保留记忆，二十年后的入门选拔上，一个带着前世记忆的
少女重新站在了他们面前。

**主要角色**：许丹青（复仇者）、林霜（前道侣/现掌门夫人）、
 苏婉儿（前师妹/现丹堂长老）、墨渊（神秘丹炉器灵）

**结局方向**：许丹青最终以丹道证道，不是杀死他们，
 而是让他们亲眼看着自己建立的丹道帝国一步步崩塌。

**推荐风格**：爽文快节奏风、情感催泪风
```

## 五、02_风格模板.md

```markdown
# 写作风格选择

## 以下 5 种预设风格，可选择一种为主风格，可选一种为辅风格叠加

---

### 【风格 1】情感催泪风

**核心追求**：让读者哭

**写作法则**：
1. 内心独白优先于外部描写。重要的不是发生了什么，是角色对此的感受。
2. 细节堆叠情绪。用具体物象承载情感：「她每天还是泡两杯茶，一杯放对面。三个月了，茶凉了倒，倒了再泡。」
3. 对话留白。人物说出口的永远只有真实想法的30%。
4. 节拍控制：铺垫(60%)→积压(20%)→瞬间爆发(10%)→余韵(10%)
5. 最关键的情感场景，描写要慢下来。时间像凝固了一样。

**禁用词**：悲伤、痛苦、愤怒、绝望、心碎（不要直接说情绪，要写出来让人感受到）

**名场面参考**：你的角色终于见到了背叛她的人。不是因为恨，而是因为对方完全不记得她了。

---

### 【风格 2】文化底蕴风

**核心追求**：读之有物，字里行间有见识

**写作法则**：
1. 世界观设定背后有真实的参照体系（历史/物理/哲学/宗教/民俗）
2. 每章至少嵌入一个「知识点」，但必须自然融入叙事，不能像百科词条
3. 角色对话体现其知识背景（读过书的人说话和没读过的人不一样）
4. 用典节制，一章不超过 2 处。用典必须服务于剧情，不能为炫学而用
5. 忌文白夹杂。可以古雅但不能假装文言文。

**名场面参考**：修仙世界里提到「灵气循环」时，用自然界的碳循环做类比，
 不说教，读者读完会觉得自己学到了一些东西。

---

### 【风格 3】爽文快节奏风

**核心追求**：每一章都有爽点，读者停不下来

**写作法则**：
1. 每章至少一个「爽点」（打脸/逆转/隐藏实力揭露/敌人吃瘪）
2. 短句为主。动作描写干净利落，环境描写≤2句
3. 章节结尾必须留钩子：「她正要离开，身后传来一个熟悉的声音...」
4. 对话推进剧情，减少大段内心独白和哲学思辨
5. 配角的作用是衬托主角，不要喧宾夺主

**禁用**：超过 200 字的纯描写段落、超过 500 字的内心独白

**名场面参考**：所有人都以为主角是废物，然后她抬手间碾压了全场最强的对手。
 不拖，一章内完成反转。

---

### 【风格 4】悬疑诡异风

**核心追求**：读者一直想知道「然后呢」「为什么」

**写作法则**：
1. 关键信息留白。暗示 > 明说，疑惑 > 解答
2. 环境渲染优先于人物：「房间很大」不如「脚步声在空房间里回荡了三秒才消失」
3. 正常中的异常最能制造悬疑感（越日常的场景越让人不安）
4. 多视角控制信息差（读者知道A不知道B，B不知道C）
5. 每章结尾留一个未解之谜
6. 谜底揭晓必须有铺垫，不能从天而降

**禁用**：上帝视角的真相解释、谜底揭晓后不再有新谜题

**名场面参考**：主角发现自己在重复过同一天。但每次重复，有些细节在改变。
 是谁在改变它们？为什么要改变？

---

### 【风格 5】轻松吐槽风

**核心追求**：读着笑出声，但不降智

**写作法则**：
1. 主角内心 OS 要有网感，接地气
2. 日常场景插入反差笑点（严肃场景里的搞笑细节）
3. 对话有来有回，节奏感强
4. 配角可以有喜剧人设但不能只靠配角搞笑
5. 关键时刻可以正经，反差更显深度
6. 笑点密度：每 500 字至少一个

**禁用**：尬笑、冷笑话、谐音梗（除非角色人设就是如此）、脱离剧情的强行搞笑

**名场面参考**：
「师尊说修行要清心寡欲。」
「那他洞府里为什么藏了三百坛酒？」
「那是悟道的媒介。」
「那我也要悟道。」
「你根基不够。」
「根基不够不就是酒量不好吗？」

---

## 风格叠加规则

如果用户同时选择两种风格：

1. 主风格（70%）：决定叙事的整体调性、节奏、读者情绪体验
2. 辅风格（30%）：在主风格基础上增加层次感

示例组合：

| 主 + 辅 | 效果 | 类似作品 |
|---------|------|---------|
| 情感催泪 + 文化底蕴 | 有文化内涵的催泪作品 | 《琅琊榜》 |
| 爽文快节奏 + 轻松吐槽 | 又爽又搞笑的爽文 | 《大王饶命》 |
| 悬疑诡异 + 文化底蕴 | 有知识厚度的悬疑 | 《古董局中局》 |
| 情感催泪 + 悬疑诡异 | 让人又哭又怕 | 《看不见的客人》 |
| 轻松吐槽 + 情感催泪 | 笑中带泪 | 《请回答1988》 |
```

## 六、03_大纲模板.md

```markdown
# 大纲生成模板

## 输入
- 用户选择的概要
- 用户选择的风格
- 用户选择的篇幅（短篇/长篇/具体章数）

## 短篇模式（5-20章）

直接生成分章大纲，每章一句话概要。

输出格式：
```
第1章：XXXXX（核心情节一句话）
第2章：XXXXX
...
第N章：XXXXX
```

## 长篇模式（50章以上）

按「卷」组织。每卷约 30-60 章。

输出格式：

```
全书分为 N 卷

【第一卷】卷标题
  主题：本卷的核心矛盾是什么
  弧线：主角从 XX 状态 → XX 状态
  关键事件：
    第1-15章：XXXX
    第16-30章：XXXX
    第31-45章：XXXX
  卷末状态：本卷结束时人物在哪、关系如何

【第二卷】卷标题
  ...
```

## 系统指令补充

1. 每卷的章数要合理（不要出现一卷只有3章或一卷100章）
2. 卷和卷之间要有自然的转折点（地点变化、时间跳跃、身份转换）
3. 每卷末尾留悬念或钩子，让读者想看下一卷
4. 大纲中标注「关键伏笔点」（哪些章埋重要伏笔）
5. 风格约束：大纲里也能体现风格调性
```

## 关系网

- 与其他角色的关系：XXX

## 出场时间表（创作过程中动态更新）
- 第X章首次出场

## 数量建议

| 篇幅 | 主角 | 重要配角 | 次要配角 |
|------|------|---------|---------|
| 短篇(5-20章) | 1-2人 | 2-3人 | 若干 |
| 长篇(50-200章) | 2-3人 | 4-6人 | 按需 |
| 超长篇(200章+) | 2-3人 | 6-10人 | 按需 |

## 风格约束

人物设定要体现风格特征：
- 情感催泪风 → 人物必须有情感软肋和遗憾
- 文化底蕴风 → 人物必须有知识背景和见识差异
- 爽文快节奏风 → 人物必须有明确的成长阶梯
- 悬疑诡异风 → 至少1个角色有隐藏身份/秘密
- 轻松吐槽风 → 至少1个角色承担笑点输出

## 八、05_细纲模板.md

```markdown
# 细纲 + 章节卡生成模板

## 输入
- 已确认的大纲
- 已确认的人物设定
- 已选择的风格

## 细纲生成

为当前卷的每一章生成细纲：

```
【第N章】章节标题
  核心情节：一句话概括本章内容
  出场人物：谁会出现
  场景：在哪里发生
  情感基调：紧张/轻松/悲伤/悬疑...
  本章目标：推进主线？发展支线？揭示伏笔？
```

## 章节卡生成

为每一章生成详细的章节卡。这是创作系统的**核心文件**。

```markdown
--- 章节卡模板 ---

# 第 N 章 章节卡

## 本章目标

## 强制输出格式

每次章节生成必须输出以下两部分，不允许省略：

```
【正文】
{实际章节内容，字数 {word_count}}

---METADATA---

【元数据块】
{
  "章节": {N},
  "埋下的伏笔": [
    {"id": "NEW-001", "内容": "玉佩在月光下出现了第二道裂纹"},
    {"id": "NEW-002", "内容": "神秘老者在城门口等的人其实是配角B"}
  ],
  "推进的伏笔": [
    {"id": "F015", "推进方式": "主角发现玉佩和古籍中的图案吻合"},
    {"id": "F028", "推进方式": "反派提到了当年的实验"}
  ],
  "回收的伏笔": [
    {"id": "F003", "回收方式": "揭露配角是卧底，对应第8章的可疑行踪"}
  ],
  "角色状态变化": [
    {"角色": "主角A", "位置": "青云宗·丹房", "状态": "刚突破筑基三层，体内暗伤未愈"},
    {"角色": "配角B", "位置": "黑风岭→青云宗", "状态": "脱离追杀，正在赶往青云宗求援"}
  ],
  "本章摘要": "主角在丹房突破时发现玉佩异变，同时配角B逃出黑风岭。二人线索开始交汇。",
  "结尾状态": "悬念（主角发现玉佩裂痕对应古籍中的末日预言）"
}
```

## 元数据块规则

1. **必须输出 JSON 格式的元数据块**，不允许省略
2. `埋下的伏笔`：本章新增的伏笔，每条给唯一ID（NEW-XXX），格式必须能录入数据库
3. `推进的伏笔`：本章推进但未回收的已有伏笔，需引用原ID
4. `回收的伏笔`：本章完全回收的伏笔，需引用原ID + 说明回收方式
5. `角色状态变化`：仅列本章中状态发生变化的角色，含位置/心理/关系/能力
6. `本章摘要`：不超过100字，用于下一章的前情缓存
7. `结尾状态`：悬念/收束/过渡，用于决定下一章的衔接方式

## 创作后自动执行

工作流解析元数据块 JSON，自动执行以下操作：

### 1. 存入前情摘要
```
读取 「本章摘要」→ 存入当前章节的前情缓存
下一章加载时自动填入【前情摘要】
```

### 2. 更新伏笔时间线（数据库）
```
读取「埋下的伏笔」→ INSERT 到活跃伏笔表，分配永久ID
读取「推进的伏笔」→ UPDATE 对应记录的推进状态
读取「回收的伏笔」→ UPDATE 状态为「已回收」，移动到归档区
```

### 3. 更新配角状态表（数据库）
```
读取「角色状态变化」→ UPSERT 配角状态表的对应记录
```

### 4. 标记章节卡完成
```
第 {N} 章章节卡 → 标记为「已完成」
```

## 字数控制

| 类型 | 每章字数 |
|------|---------|
| 短篇 | 2000-3000字/章 |
| 长篇标准 | 3000-4000字/章 |
| 长篇详细 | 4000-6000字/章 |
| 网文日更 | 2000-2500字/章 |

用户可在创作前指定每次的字数。
```

|------|---------|---------|-------------|

## 自动维护规则

每次生成新章节后：
1. 检测本章出场的角色 → 更新当前位置和状态
2. 检测角色是否有重大状态变化（受伤/升级/位置转移/关系改变）→ 更新
3. 如有角色退场（死亡/离开主线）→ 移动到「已退场角色」
4. 新角色首次出场 → 从人物MD中提取信息添加

## 加载规则

创作新章节时，只加载「活跃角色」。
不加载「已退场角色」。

## 关键：角色在章节卡中未出场 ≠ 退场

很多角色可能几章不出场但仍然活跃。
例如：配角B 第70章没出现，但他在第71章会出现。
→ 他的状态保持为「正在前往主角的路上」而不是删除。
→ 直到剧情明确他退场（死亡/离开/故事线结束），才归档。
