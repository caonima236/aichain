# 长篇创作工坊 — 独立工作流拆分

> 每个工作流独立上传到 Coze，通过主 Bot 调度拼接。
> 数据通过 Coze 数据库 + 知识库共享。

---

## 工作流总览

```
用户输入世界观
    │
    ▼
【WF1_概要生成】 ─── 输出：3个概要
    │
用户选择概要+风格+篇幅
    │
    ▼
【WF2_大纲生成】 ─── 输出：分卷大纲
    │
用户确认
    │
    ▼
【WF3_人物生成】 ─── 输出：人物MD
    │
用户确认
    │
    ▼
【WF4_章节卡生成】 ─ 输出：第一卷细纲+章节卡
    │
    │  ◄────────── 循环 ──────────┐
    ▼                              │
【WF5_创作章节】                     │
  ├── 加载上下文                    │
  ├── 生成正文+元数据JSON            │
  └── 调用 WF6_入库                 │
    │                              │
    ▼                              │
 继续/下一章 ───────────────────────┘
    │
    ▼
【WF7_卷切换】 ─── 归档当前卷 → 生成新卷章节卡
    │
    ▼
 回到 WF5（继续创作）

【WF8_进度面板】 ─ 随时查询
```

## WF1_概要生成

```yaml
工作流名称: 长篇创作_概要生成
触发方式: Bot调用 / 手动
输入参数:
  - world_setting: string（用户输入的世界观+剧情方向）
输出:
  - summary_a: string（概要A：爽文路线）
  - summary_b: string（概要B：悬疑复杂路线）
  - summary_c: string（概要C：情感深度路线）

节点编排:
  [开始] → [分析世界观] → [生成概要A] ┐
                          → [生成概要B] ├→ [合并输出] → [结束]
                          → [生成概要C] ┘

提示词模板:
---
你是一个专业的故事策划。请根据以下世界观和剧情方向，
生成三个不同路线的故事概要，每个约300字。

【世界观输入】
{world_setting}

【概要A：爽文路线】
核心：冲突密度高、节奏快、每阶段爽点清晰
请包含：故事标题、一句话卖点、核心冲突+主线走向、
主要角色构成、结局方向

【概要B：悬疑复杂路线】  
核心：真相层层剥开、反转多、伏笔密、敌我难辨
请包含：同上结构

【概要C：情感深度路线】
核心：人物关系复杂、情感冲突驱动剧情、有代价有牺牲
请包含：同上结构
---
```

## WF2_大纲生成

```yaml
工作流名称: 长篇创作_大纲生成
输入参数:
  - selected_summary: string（用户选择的概要全文）
  - style_primary: string（主风格：「情感催泪」「文化底蕴」「爽文快节奏」「悬疑诡异」「轻松吐槽」）
  - style_secondary: string（辅风格，可为空）
  - chapter_count: number（总章数）
输出:
  - outline_markdown: string（分卷大纲Markdown）

节点编排:
  [开始] → [加载风格定义] → [生成分卷大纲] → [结束]

提示词模板:
---
你是一个专业的小说结构师。根据已确认的故事概要和写作风格，
生成分卷大纲。

【已确认概要】
{selected_summary}

【写作风格】
主风格：{style_primary}
辅风格：{style_secondary}

【篇幅要求】
总章数：约 {chapter_count} 章

【风格约束】
{从知识库加载对应风格定义}

【输出要求】
全书分为若干卷，每卷30-60章。

按以下格式输出：

全书分为 N 卷

【第一卷】卷标题
  主题：（本卷的核心矛盾）
  弧线：主角从XX状态→XX状态
  关键事件：
    第1-15章：XXXX
    第16-30章：XXXX
    ...
  卷末状态：本卷结束时人物在哪、关系如何

...（后面各卷同理）

要求：
1. 每卷章数合理，不出现极端值
2. 卷与卷之间自然转折
3. 每卷末尾留钩子
4. 标注关键伏笔点
---
```

## WF3_人物生成

```yaml
工作流名称: 长篇创作_人物生成
输入参数:
  - outline: string（已确认的大纲全文）
  - style_primary: string
  - style_secondary: string
  - chapter_count: number
输出:
  - characters_json: string（人物列表JSON，每个角色为独立对象）

节点编排:
  [开始] → [解析大纲提取角色线索] → [逐个生成角色] → [合并输出JSON] → [结束]

提示词模板:
---
你是一个专业的人物设计师。根据大纲和风格，为故事创建角色。

【大纲】
{outline}

【风格要求】
主风格：{style_primary}
辅风格：{style_secondary}

【风格对人物的约束】
- 情感催泪风：每个主要角色必须有情感软肋和未完成的遗憾
- 文化底蕴风：角色的知识背景和见识要有差异
- 爽文快节奏风：角色必须有明确的成长阶梯
- 悬疑诡异风：至少1个角色有隐藏身份/秘密
- 轻松吐槽风：至少1个角色承担笑点输出

【篇幅建议】
{chapter_count}章——主角{主角数量}人，重要配角{配角数量}人

【输出格式】
为每个角色输出JSON对象，集合为数组：

[
  {
    "姓名": "XXX",
    "类型": "主角/重要配角/反派",
    "年龄": XX,
    "身份": "XXX",

      "出身": "XXX",
      "关键事件": ["事件1", "事件2"],
      "深层动机": "TA真正想要什么"
    },
    "语言风格": {
      "说话特征": "口头禅/语速/句式",
      "情绪表达": "外向/内敛/压抑"
    },
    "成长弧线": {
      "初始状态": "XXX",
      "转折点": "XXX",
      "最终状态": "XXX"
    },
    "关系网": {
      "与主角关系": "XXX",
      "与其他角色": "XXX"
```

## WF5_创作章节（核心引擎）

```yaml
工作流名称: 长篇创作_创作章节
输入参数:
  - chapter_number: number
  - project_id: string
输出:
  - chapter_content: string（正文）
  - metadata_json: string（元数据JSON）

节点编排:
  [开始]
    → [从数据库加载：全书概要+当前卷细纲]
    → [从数据库加载：前情摘要缓存]
    → [从数据库加载：当前章节卡]
    → [从数据库加载：活跃伏笔列表]
    → [从数据库加载：配角当前状态]
    → [从知识库加载：风格定义]
    → [组装上下文提示词]
    → [调用LLM生成章节]
    → [分离正文和元数据]
    → [输出正文+元数据JSON]
    → [调用WF6_入库(metadata_json)]
    → [结束]

提示词模板:
---
你是一个专业的小说作家。创作第 {chapter_number} 章。

【全书概要】
{project_summary}

【当前卷背景】
{volume_context}

【前情摘要】（上一章发生了什么）
{previous_summary}

【本章章节卡】
{chapter_card}

【人物当前状态】
{character_states}

【活跃伏笔清单】（必须在后续章节推进或回收）
{active_threads}

【创作要求】
1. 严格遵循章节卡规划的场景流程
2. 人物说话和行为必须符合人物设定
3. 本章需推进的伏笔：{推进列表}
4. 本章需回收的伏笔：{回收列表}——必须在本章完成回收
5. 新伏笔在元数据块中声明
6. 字数：3000-4000字
7. 风格：{style_primary} + {style_secondary}

【风格详细定义】
{style_definition}

【强制输出格式】
你的输出必须严格分为两部分：

【正文】
{实际章节内容}

---METADATA---

【元数据块】
{
  "章节": {chapter_number},
  "埋下的伏笔": [
    {"id": "", "内容": ""}
  ],
  "推进的伏笔": [
    {"id": "", "推进方式": ""}
  ],
  "回收的伏笔": [
    {"id": "", "回收方式": ""}
  ],
  "角色状态变化": [
    {"角色": "", "位置": "", "状态": ""}
  ],
  "本章摘要": "不超过100字",
  "结尾状态": "悬念/收束/过渡"
}

注意：
- 如果本章没有新伏笔，「埋下的伏笔」输出空数组 []
- 如果本章没有推进任何伏笔，「推进的伏笔」输出空数组 []
- 如果本章没有回收任何伏笔，「回收的伏笔」输出空数组 []
- 角色状态变化只列实际发生变化的角色
- 本章摘要用于下一章的前情缓存
---
```

## WF6_元数据入库

```yaml
工作流名称: 长篇创作_元数据入库
输入参数:
  - metadata_json: string（WF5输出的元数据JSON）
  - project_id: string
  - chapter_number: number
输出:
  - status: string（"success"/"error"）

节点编排:
  [开始]
    → [解析JSON]
    → [更新前情摘要缓存：写入「本章摘要」]
    → [处理伏笔：
        ├── 「埋下的伏笔」→ INSERT到活跃伏笔表，分配永久ID
        ├── 「推进的伏笔」→ UPDATE状态
        └── 「回收的伏笔」→ UPDATE为「已回收」+归档]
    → [处理角色状态：
        └── 「角色状态变化」→ UPSERT配角状态表]
    → [标记章节完成：更新章节索引表]
    → [返回状态]

数据库操作（伪代码）:
---
// 1. 更新前情摘要
UPDATE project_cache SET previous_summary = metadata.本章摘要 WHERE project_id = {project_id}

// 2. 新增伏笔
FOR EACH 伏笔 IN metadata.埋下的伏笔:
  INSERT INTO foreshadowing (project_id, content, buried_chapter, status)
  VALUES ({project_id}, 伏笔.内容, {chapter_number}, '活跃')
  RETURNING id → 可返回给用户知道分配了什么ID

// 3. 更新推进的伏笔
FOR EACH 伏笔 IN metadata.推进的伏笔:
  UPDATE foreshadowing SET 
    last_advanced_chapter = {chapter_number},
    advancement_note = 伏笔.推进方式
  WHERE id = 伏笔.id

// 4. 回收伏笔
FOR EACH 伏笔 IN metadata.回收的伏笔:
  UPDATE foreshadowing SET
    status = '已回收',
    resolved_chapter = {chapter_number},
    resolution_note = 伏笔.回收方式
  WHERE id = 伏笔.id

// 5. 更新角色状态
FOR EACH 变化 IN metadata.角色状态变化:
  UPSERT character_status (project_id, character_name, location, status, last_chapter)
  VALUES ({project_id}, 变化.角色, 变化.位置, 变化.状态, {chapter_number})

// 6. 标记章节完成
UPDATE chapter_index SET status = '已完成', completed_at = NOW()
WHERE project_id = {project_id} AND chapter_number = {chapter_number}
---
```

## WF7_卷切换

```yaml
工作流名称: 长篇创作_卷切换
输入参数:
  - project_id: string
  - new_volume: number
输出:
  - new_chapter_cards_json: string
  - status: string

节点编排:
  [开始]
    → [归档当前卷：
        ├── 当前卷章节卡标记「已归档」
        ├── 已回收伏笔移到归档区
        └── 已退场角色标记「已退场」]
    → [环境快照：
        保存核心人物状态+休眠伏笔+世界观状态]
    → [调用WF4_章节卡生成(new_volume)]
    → [重置前情摘要缓存]
    → [输出新卷章节卡]
    → [结束]

提示词模板:
---
当前第 {current_volume} 卷已完成。启动第 {new_volume} 卷。

【上一卷结束时状态】
{volume_end_snapshot}

【需要在第 {new_volume} 卷推进的休眠伏笔】
{dormant_threads}

【第 {new_volume} 卷大纲】
{next_volume_outline}

请以「新开始」的状态创作——
只保留全书概要+人物核心设定+活跃伏笔，
旧卷的详细章节卡不再加载。
新卷上下文窗口回到 ~2000 字。
---
```

## 主 Bot 调度逻辑

```yaml
Bot名称: 长篇创作工坊
人设: 
  你是长篇创作助手。帮助用户从世界观到成稿完成整本书创作。
  使用工作流拼接实现分步创作。

意图路由:

用户说「开始创作」「新书」「我想写...」:
  → 收集世界观 → 调用 WF1_概要生成
  → 展示3个概要 → 等待选择

用户说「选第X个」「选概要A」:
  → 确认概要 → 问风格 → 问篇幅
  → 调用 WF2_大纲生成
  → 调用 WF3_人物生成
  → 调用 WF4_章节卡生成(volume=1)
  → 提示：「准备完成！输入『写第1章』开始创作」

用户说「写第N章」「继续」「下一章」:
  → 调用 WF5_创作章节(chapter=N)
  → WF5内部自动调用WF6入库
  → 展示正文

用户说「进度」「状态」「面板」:
  → 调用 WF8_进度面板

用户说「进入下一卷」「开始第X卷」:
  → 调用 WF7_卷切换 → 完成后提示继续
```
